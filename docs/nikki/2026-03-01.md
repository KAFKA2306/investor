# 2026-03-01 開発日記：非伝統的データによる直交アルファの「自律抽出」と「先行指標（Lead-Lag）」への拡張

本日、EDINETの開示情報をトリガーとした「ガバナンス脆弱性アルファ」の抽出・検証スキームを完遂し現在は、2021年以降のデータに基づくパフォーマンス指標の再計算を行っている。

### 追記：実証の深化と統計的堅牢性 (09:15)

実証プロセスをさらに進める中で、以下の統計的知見を得た：

1.  **IC（インフォメーション・コーリレーション）の安定化**:
    少数のシグナル下では順位相関（Gauss Rank IC）が不安定になり、`NaN`を誘発するリスクがある。計算ロジックにガードを設けつつ、ユニバースを16銘柄に倍増させることで、シグナルの分散を確保し、評価の信頼性を向上させた。
2.  **情報の非対称性と時間的ラグ**:
    10-K（有価証券報告書）の公示から市場価格への浸透には時間的なラグが存在する。このラグをアルファとして捉えるため、PIT（Point-In-Time）に基づいた遅延反映ロジックの妥当性を、過去の価格データとの統合によって検証中である。
「Macro-to-Micro Lead-Lag」探索へと拡張しました。

## 追記：J-Quants 5年キャッシュ拡張（cache優先・no fallback/no retry）

`latest 5y` の価格キャッシュを、**「キャッシュがあれば必ず使う」**方針で前進。
運用ルールは以下に固定した。

- `cache first`: 既存キャッシュを最優先で利用
- `never fallback`: 代替APIや別経路にフォールバックしない
- `never retry`: 同一失敗リクエストを即時リトライしない
- `date-mode`: 1日×全銘柄で密度高く取得
- `desc`: 最新日から埋める

### 実行ログ（要約）
- 1st chunk（2025-12-01〜12-05）:
  - `requests=5`, `cacheHits=2`, `network=3`, `rateLimit=0`, `rows=22,138`
- 2nd chunk（2025-11-24〜11-28）:
  - `requests=5`, `cacheHits=0`, `network=5`, `rateLimit=0`, `rows=17,704`
- 3rd chunk（2025-11-20〜11-21付近）:
  - `requests=2`, `rateLimit=2`, `abortedByRateLimit=true`, `rows=0`
  - 連続429検知で安全停止（5分クールダウン前提）

### 運用改善（Taskfile化）
J-Quants の全期間キャッシュ取得を、毎回の手打ち実行から `Taskfile.yml` タスク運用へ切り替えた。

- 追加タスク:
  - `task jquants:warm-all:start`（完了までループ）

  - `task jquants:warm-all:log`（ログ確認）
  - `task jquants:warm-all:stop`（停止）
- 実装ポリシー:
  - `--mode=date`（最小リクエスト）
  - `--req-per-30sec=1`

  - 銘柄カバレッジ（1営業日あたり実測）: 約 `4,418` 〜 `4,429` 行（全銘柄日次バー）

  - 接続失敗時は `60s` 待機して再試行
- ログ/PID:
  - log: `/tmp/jquants_warm_until_finish.log`
  - pid: `/tmp/jquants_warm_until_finish.pid`
